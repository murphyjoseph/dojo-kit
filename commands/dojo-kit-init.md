# dojo-kit init

Generate a `dojo-kit.yaml` configuration file by scanning the current project. This file describes the project's stack, structure, and conventions so that dojo-kit skills have context when generating code.

Read the YAML schema at `${CLAUDE_PLUGIN_ROOT}/docs/references/dojo-kit-yaml-schema.md` before generating the file.

## Steps

### 1. Detect project basics

Scan the repo root to determine:

- **Package manager** — check for lockfiles: `pnpm-lock.yaml` → pnpm, `package-lock.json` → npm, `yarn.lock` → yarn, `bun.lockb` → bun
- **Project type** — check for workspace config: `pnpm-workspace.yaml`, `workspaces` in package.json → monorepo; otherwise single-app
- **Framework** — check `package.json` dependencies: `next` → Next.js, `@remix-run/*` → Remix, `astro` → Astro, `vite` + no framework → Vite SPA
- **Language** — check for `tsconfig.json` → TypeScript, otherwise JavaScript

### 2. Detect libraries

Scan `package.json` (and workspace root if monorepo) for known libraries:

| Category | Look for |
|---|---|
| State management | `zustand`, `@reduxjs/toolkit`, `jotai`, `valtio` |
| Data fetching | `@tanstack/react-query`, `swr`, `@apollo/client`, `urql` |
| Forms | `react-hook-form`, `formik`, `@tanstack/react-form` |
| Validation | `zod`, `yup`, `valibot`, `@effect/schema` |
| Styling | `tailwindcss`, `styled-components`, `@emotion/*`, `vanilla-extract` |
| Testing | `vitest`, `jest`, `@testing-library/*`, `playwright`, `cypress` |
| UI library | `@radix-ui/*`, `@headlessui/*`, `@shadcn/ui`, `@mantine/*` |

### 3. Detect monorepo structure (if applicable)

If the project is a monorepo:

- List all workspace packages from the workspace config
- For each package, read its `package.json` and infer its type:
  - Has `next`/`remix`/`astro` → app
  - Has `react` as peer dep → react library
  - Has no react → platform/tooling library
- Group packages by type

### 4. Ask about ambiguity

Use `AskUserQuestion` for things that can't be inferred. Maximum 5 questions. Common questions:

- "What CSS approach does this project use?" (if multiple styling deps found)
- "Which of these apps is the primary one?" (if multiple apps in monorepo)
- "Does this project use server-side rendering?" (if framework supports both)
- "What testing strategy do you follow?" (unit, integration, e2e, or combination)

Skip questions where the answer is clear from the scan.

### 5. Check for existing dojo-kit.yaml

If `dojo-kit.yaml` already exists:

- Read it
- Skip questions that are already answered
- Only ask about fields that are missing or set to `~` (undecided)
- Merge new detections with existing values (don't overwrite explicit choices)

### 6. Generate dojo-kit.yaml

Write the file following the schema exactly. Use these conventions:

- **Present value** = decided (e.g., `packageManager: pnpm`)
- **`~` (null)** = undecided, will be resolved later
- **Absent key** = not applicable to this project

Include a comment at the top: `# Generated by dojo-kit init — edit freely`

### 7. Generate or audit CLAUDE.md

Check if a `CLAUDE.md` exists at the project root.

**If no CLAUDE.md exists:** Ask the user if they'd like one generated. If yes, use the templates from `${CLAUDE_PLUGIN_ROOT}/skills/claude-md-improver/references/templates.md` to create one. Choose the template that matches the project type detected in earlier steps (minimal, comprehensive, monorepo, or package). Populate it with real values from the scan — actual commands from `package.json` scripts, the detected directory structure, and any gotchas discovered during detection (e.g., specific lockfile, workspace config quirks). **Always include the architecture section** from `${CLAUDE_PLUGIN_ROOT}/docs/references/claude-md-architecture-section.md` — this ensures file organization patterns are always in context, not just when skills trigger. Write the file.

**If CLAUDE.md already exists:** Check if it contains an `## Architecture (dojo-kit)` section. If not, ask the user if they'd like the dojo-kit architecture rules added to their CLAUDE.md. If yes, read the architecture section from `${CLAUDE_PLUGIN_ROOT}/docs/references/claude-md-architecture-section.md` and append it to the existing CLAUDE.md. If the section already exists, skip — don't duplicate it.

**Why this matters:** Plugin skills only load on demand. The architecture rules must be in CLAUDE.md (always loaded) to ensure Claude follows the file organization patterns even when skills don't trigger.

### 8. Offer companion plugins and tools

dojo-kit does not bundle third-party plugins or skills — it recommends and installs them on your behalf. After generating `dojo-kit.yaml` and `CLAUDE.md`, present the companion catalog using `AskUserQuestion` (multi-select).

**Always ask about these companions:**

| Companion | Type | What it provides |
|---|---|---|
| `superpowers` (Anthropic) | Plugin | Brainstorming, TDD, plan execution, code review workflows |
| `frontend-design` (Anthropic) | Plugin | Production-grade UI design — typography, color, motion, spatial composition |
| `context7` (Upstash) | MCP server | Up-to-date library documentation via Context7 |
| `typescript-lsp` | LSP server | TypeScript/JavaScript language server for diagnostics |

**Conditional companions (suggest when detected):**

| Companion | Condition |
|---|---|
| `nextjs` (Anthropic) | Next.js detected in dependencies |

**Question format:**

Use `AskUserQuestion` with `multiSelect: true`:

> "dojo-kit works well with these companion tools (all free, all optional). Which would you like to install?"

List each companion with a short description. Pre-select companions that match the detected stack (e.g., `typescript-lsp` when TypeScript is detected, `context7` always recommended).

**Installation:**

For each selected companion:

- **Plugins** (`superpowers`, `frontend-design`, `nextjs`) → Add to the project's `.claude/settings.json` under `enabledPlugins` (e.g., `"superpowers@claude-plugins-official": true`). If `extraKnownMarketplaces` doesn't already include `claude-plugins-official`, add it with `"source": "github", "repo": "anthropics/claude-code"`. Create `.claude/settings.json` if it doesn't exist; merge if it does.
- **MCP servers** (`context7`) → Write the config to the project's `.mcp.json` file (create if absent, merge if exists)
- **LSP servers** (`typescript-lsp`) → Write the config to the project's `.lsp.json` file (create if absent, merge if exists)

Tell the user to restart their Claude Code session for plugin changes to take effect.

**Record in dojo-kit.yaml:**

Add a `companions:` section to the generated `dojo-kit.yaml` recording what was installed:

```yaml
companions:
  plugins:
    - superpowers
    - frontend-design
  mcp:
    - context7
  lsp:
    - typescript-lsp
```

On re-run, skip companions that are already recorded in `dojo-kit.yaml` — only ask about new ones.
