# 002 — Project Config is a Generated YAML

## Decision

Each project gets a `dojo-kit.yaml` generated by the `/dojo-kit-init` command. The YAML is the output of initialization, not a template to fill in.

## Config Shape

- `project` — type (monorepo/single-app), package manager (detected from lockfile), framework
- `apps` — monorepo only, one entry per app with framework and libraries scoped to it
- `libraries` — cross-boundary or single-app libraries with three states:
  - Value present → decided
  - `~` (null) → needed but undecided, Claude should ask before choosing
  - Key absent → not applicable
- `packages` — monorepo only, each with a name and type (`platform`, `react`, `tooling`)
- `docs` — location and tracked sections
- `servers` — MCP/LSP servers to configure

## Init Behavior

- Existing project → scan deps/lockfile/structure, fill what's certain, ask about ambiguity
- New project → ask structural questions upfront, generate from answers
- Re-running init reads existing YAML and skips resolved questions

## Package Types

| Type | Meaning | Rule |
|---|---|---|
| `platform` | Pure TS/JS, zero framework deps | Must not import React, framework code, etc. |
| `react` | React-specific utilities | React as peer dep, clearly separated from platform |
| `tooling` | Build/dev only | ESLint configs, tsconfigs, scripts |
